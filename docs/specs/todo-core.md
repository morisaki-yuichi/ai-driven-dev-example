# 仕様 001 — TODO ドメインコア

> 仕様駆動の題材（第2部）。受け入れ条件を先に決め、これをプロンプトの原材料にする。
> 実装 `app/todo/core.py` / テスト `app/todo/test_core.py`。

## ユーザーストーリー

利用者として、やることを登録し、一覧で確認し、終わったら完了にしたい。

## スコープ

純粋なドメインロジックのみ（Web・永続化なし）。後続スライスで API／永続化を足す。

## 受け入れ条件（＝合否はこれで判定。出力の一致では判定しない）

- **AC1 追加**: 空でないタイトルで追加すると、未完了（`completed=False`）の Todo が1件増え、
  `id` は 1 からの連番。
- **AC2 バリデーション**: タイトルは前後の空白を trim。空白のみ／空は追加不可（`ValueError`）。
- **AC3 一覧**: 追加順で返る。
- **AC4 完了**: `complete(id)` で該当 Todo が `completed=True`。存在しない `id` は `KeyError`。

## 検証網への接続

上の各 AC は `app/todo/test_core.py` の1テストに対応し、pytest → CI で恒久的に守られる。
これが「受け入れ条件を検証網に接続する」の実体（Sprint 1 Try② の回収）。
