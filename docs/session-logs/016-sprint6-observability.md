## 実装のポイント

- 集計は**純粋関数**（`load_records` / `summarize` / `format_report`）＋薄い `main`。テストしやすさ優先。
- 実トレースは async 追記で**壊れた行が混じりうる**ので、パースは行単位で頑健に（壊れた行は数えて飛ばす）。

## 判断（採用/棄却）

| 判断点 | 採用 | 理由 | 棄却 |
|--------|------|------|------|
| 時間の測り方 | 隣接 `ts` の**ステップ経過** | hook 1本で取れる・「どこに時間が寄るか」は十分見える | Pre/Post ペアで単体レイテンシ（hook 2本・相関付けが要る→Try） |
| ラベルの正直さ | 「ツール＋モデル＋待ち込み」と明記 | 近似を単体時間と偽らない（誤った最適化を防ぐ） | 「ツール所要」と称する（測っていないものを名乗る） |
| トークン/コスト | **今回は出さない**（Try に送る） | hook 入力に無い。推測で埋めない（知識の鮮度） | それらしい数字を捏造（事故） |
| ts 精度 | 秒→ミリ秒に引き上げ | 時間分析に秒は粗い。混在は集計側で吸収 | 秒のまま（遅いステップが潰れる） |
| 壊れた行 | 数えて飛ばす | async 追記の現実に頑健。落とさず可視化 | 例外で停止（1行で集計全滅） |
| 集計器の置き場 | `tools/` を gate に追加 | ロードマップ指示「検証網に載せる」を満たす | 素の実装のみ（回帰で静かに壊れる） |

## 検収観点

- 「遅い」の体感が、集計で**具体的なステップ**に変換されているか（fixture の `#3 pytest`）。
- 測っているもの（ステップ経過）を正しく名指しているか（単体レイテンシと偽っていない）。
- 一本道（hook → JSONL → 集計）が実トレースで通っているか。
