## 2. 全編を貫いた原則

1. AI の出力はそのままでは信頼しない → **検証網**で受ける（正しさ）。
2. 操作の被害範囲を絞る → **檻＋最小権限**（ブラスト半径）。
3. **会話を信頼の源泉にしない。決定はファイルへ**（000〜014 が実証）。
4. **知識の鮮度**: 記憶で書かず実測（MCP SDK の introspection/probe、バージョン固定）。
5. **実行して観察**: 実験の観察は必ず本物の出力。再構成なら明記。
6. **PR → CI 緑 → マージ → main 確認**、1コミット=1意味。

## 3. AI の失敗カタログ（症状 → 原因 → 検出網/防御 → 実験）

| 症状 | 原因 | 検出網・防御 | 実験/回 |
|------|------|--------------|---------|
| 存在しない API を自信満々に使う | 幻覚 | 型チェック → 実行 → テスト（安い網から） | 実験00 |
| 破壊的操作で被害が広がる | 権限が広すぎる | 檻(sandbox/DevContainer) ＋ 最小権限 | 実験01 |
| 置き場所・言語・体裁を外す | 常時文脈(CLAUDE.md)が無い | CLAUDE.md（テストで守れない規約を担う） | 実験02 |
| 序盤の制約を忘れる | コンテキスト圧縮 | 決定はファイルへ外部化（＋圧縮時に残す指示） | 実験03 |
| ツール説明の注入命令に従いかける | 外部テキストを命令扱い | 原則「入力≠命令」＋最小権限＋監査 | 実験04 |
| 「分割すれば速い/安い」と誤信 | 固定オーバーヘッドの無視 | 時間/コストを計測して分割を設計判断に | 実験05 |
| 頼まれていない機能まで作る（過剰実装） | スコープ無限定 | 影響調査を先に書き出す | 013(priority) |
| 古い SDK/API を使う | 知識の鮮度 | 実測(introspection/probe)・公式ドキュメント参照 | 008(MCP) |
| 実装に合わせた形骸テスト | テスト後追い | テスト先行（赤→緑）で受け入れ条件を固定 | 006(仕様駆動) |
