# 概念解説 08 — 評価（eval・良くなったかを質で測る）

> 第7部。3点形式。執筆日 2026-07-05 / claude-opus-4-8。前提: 概念06「hooks」/ 概念07「可観測性」。

検証網（tests）は「実装が正しいか」を測る。だがプロンプト・ツール・ハーネスを変えたとき、
「**モデルの出力が良くなったか**」は tests では測れない。それを測る網が eval。
「良くなった気がする（vibes）」を、**再現する数字**に変える層だ。

## 1. eval（質を採点する網）

- **① 一言定義**: タスク集合に対しモデルを走らせ、出力が**期待特性**を満たすかを採点して合否を出す仕組み。
- **② なぜ必要か・事故例**: プロンプトを「改善」したつもりが**実は悪化**しても、出力の見た目は良く見え、
  vibes では気づけない。tests は実装しか見ないので素通りする。質の回帰を止める網が要る。
- **③ このリポジトリでの実例**: [`eval/`](../../eval/) が make_slug（タイトル→スラッグ）を採点。
  [実験08](../../experiments/08-eval/) では“読みやすく改善”した B が **1.000→0.571** と回帰したのを検出した。

## 2. 決定スコア（コードで採点する）

- **① 一言定義**: 期待特性を**決定的な述語**（小文字か・URL安全か・キーワードが残るか 等）で採点する方式。
- **② なぜ必要か・事故例**: 採点自体をモデルに任せる（LLM-as-judge）と、採点コストと**採点バイアス**という
  別の不確実性が乗る。まずは決定スコアで土台を作り、判定を再現可能にしておく。
- **③ このリポジトリでの実例**: [`harness.py`](../../eval/harness.py) の `CHECKS` は7つの述語。落ちた特性まで
  出す（`url_safe×6` 等）ので、スコアだけでなく**どこが壊れたか**まで分かる。LLM-as-judge は将来 Try。

## 3. 何を測り、何を測らないか（スコープと正直さ）

- **① 一言定義**: eval の**スコープ**（どのタスク・どの特性を合否にするか）と、**測っていないもの**を明示する。
- **② なぜ必要か・事故例**: 特性が不十分だと「緑なのに悪い」を見逃す。逆に測れないもの（例: 実 API のコスト/
  モデル階級比較）を測ったふりをすると事故る。モデルID・価格は推測しない（知識の鮮度）。
- **③ このリポジトリでの実例**: 採点は**本物**（収録出力に対する実計算）だが、モデル出力は API 不在のため
  収録フィクスチャで、live 未実行と明記した。価格は [`pricing.py`](../../eval/pricing.py) に実測固定し
  「Models API で確認」注記を付けた。モデル階級比較（Haiku/Sonnet/Opus の品質×時間×コスト）は将来 Try。

## まとめ

eval は「変更が良くなったか」を**質**で判定する網。tests（正しさ）・可観測性（行動/時間/コスト）の上に載る
最後の層だ。まず決定スコアで最小の土台を作り、CI に載せれば回帰を PR で止められる。守る eval 自身も
検証網に載せて守る（[`test_eval_harness.py`](../../eval/test_eval_harness.py)）。
第2版（ハーネスを作る/運用する編）の3層——**決定的な足場（hooks）→ 可観測性 → 評価**——はここで一区切り。
