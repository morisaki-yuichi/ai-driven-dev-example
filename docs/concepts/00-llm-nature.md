# 概念解説 00 — LLM の性質（なぜ検証なしで信じてはいけないか）

> 第0部の本編。執筆日 2026-07-05 / モデル claude-opus-4-8。
> 各項は3点形式：**①一言定義 ②なぜ必要か・事故例 ③このリポジトリでの実例**。

AI 駆動開発の第一歩は「AI にコードを書かせる」ことではなく、
**AI の出力が信頼できないことを前提に、信頼できる仕組みを外側に作る**ことです。
その前提を理解するために、LLM が持つ4つの性質を押さえます。

---

## 1. 非決定性（non-determinism）

- **① 一言定義**: 同じ入力（プロンプト）でも、出力が毎回変わりうる性質。
- **② なぜ必要か・事故例**: 「昨日はこのプロンプトで動くコードが出たのに、今日は別物が出る」。
  だから**追体験を「コードの一致」で測ることはできない**。教材でも写経者の手元に
  同じコードは出ない。合否は出力一致ではなく、あらかじめ決めた受け入れ条件で判定するしかない。
- **③ このリポジトリでの実例**: 追体験の単位を「コード」ではなく
  **「指示 → 出力 → 評価 → 修正指示」＋判断の注釈**（＝編集版セッションログ）にした。
  → [`docs/initial-prompt.md`](../initial-prompt.md) の「設計課題1」、実演は
  [`docs/session-logs/000-setup.md`](../session-logs/000-setup.md)。

## 2. 幻覚（hallucination）

- **① 一言定義**: 事実・API・引数などを、**もっともらしい言い回しで捏造する**性質。
  「知らない」と言う代わりに、それらしい嘘を自信満々で返す。
- **② なぜ必要か・事故例**: 存在しない関数・引数・ライブラリを使ったコードが、
  一見きれいに書かれて出てくる。レビューで素通りすると、**実行した瞬間に崩れる**。
  怖いのは「間違い」そのものより、**間違いに自信が伴う**こと。
- **③ このリポジトリでの実例**: 実験00で、存在しない `str.slugify()` を使う一次出力を
  検証網（実行・テスト）が捕まえる様子を体験する。
  → [`experiments/00-hallucinated-api/`](../../experiments/00-hallucinated-api/)。

## 3. コンテキスト窓（context window）

- **① 一言定義**: 一度に「見える」情報量に上限がある。超えると古い部分が
  要約・圧縮され、細部が脱落する。
- **② なぜ必要か・事故例**: 長い会話の**序盤で決めた制約**を、後半で忘れる。
  「さっき合意したはずの方針」と食い違う出力が、悪気なく出てくる。
- **③ このリポジトリでの実例**: 原則は**「会話を信頼の源泉にしない。決定はファイルへ」**。
  セットアップ合意を会話に残さず [`000-setup.md`](../session-logs/000-setup.md) に外部化したのが
  その最初の実演。圧縮で制約が落ちる様子と外部化の効果の比較は第2部で実験する。

## 4. 知識の鮮度（knowledge freshness）

- **① 一言定義**: 学習データにはカットオフがあり、**それ以降の API・バージョン・価格・
  モデルID を知らない**。古い知識を、現在の事実として返してしまう。
- **② なぜ必要か・事故例**: 非推奨・廃止されたAPIや、古いライブラリの書き方、
  過去のモデルIDや価格を「最新」と信じて提案する。
- **③ このリポジトリでの実例**: コード例のモデルIDは執筆時点の推奨
  `claude-opus-4-8` を明記しつつ、**「Models API で最新を確認して差し替える」**注記を必ず添える。
  価格は「執筆時点」明記の付録に置き、本文では最新を引く手順を教える。
  → [`docs/initial-prompt.md`](../initial-prompt.md) の「モデルとコストの方針」。

---

## まとめ — だから「検証網」を先に作る

4つの性質はいずれも「AI の出力は、そのままでは信頼できない」ことを示す。
だが**信頼できないことは、使えないことを意味しない**。人間の書いたコードも、
そのままでは信頼せずレビューとテストで守るのと同じで、AI の出力も
**検証網（テスト・型チェック・lint・CI）で受け止める前提**にすれば戦力になる。

第0部の実験00は、この構図を最短で体験するためにある
——「幻覚 → 実行/テストが捕まえる → 仕様を足して修正 → 再実行で緑」。
第1部以降は、この検証網そのものを設計・構築していく。
