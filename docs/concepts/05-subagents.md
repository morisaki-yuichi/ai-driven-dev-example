# 概念解説 05 — sub-agents（委譲・並列化・コスト）

> 第3部。3点形式。執筆日 2026-07-05 / claude-opus-4-8。前提: 概念03「4層の記憶モデル」。

skill が「手順の記憶」なら、sub-agent は **作業そのものの委譲**。別の文脈を持つエージェントに
タスクを預け、必要なら並列に走らせる。ただし**タダではない**。

## 1. 委譲（delegation）

- **① 一言定義**: sub-agent に作業を丸ごと預け、結果だけ受け取る。
- **② なぜ必要か・事故例**: 大きな調査を親の文脈で全部やると窓を圧迫する。委譲すれば
  親の文脈を汚さず、専門タスクを分離できる。ただし委譲先の暴走に備え権限は絞る（第1部）。
- **③ このリポジトリでの実例**: [実験05](../../experiments/05-subagents/) で調査タスクを Explore エージェントに委譲した。

## 2. 並列化（parallelism）

- **① 一言定義**: 互いに独立した部分を、複数の sub-agent で同時に走らせる。
- **② なぜ必要か・事故例**: 依存がある部分は並列化できず待ちが出る。分割が**不均衡**だと、
  一番遅い担当が全体の時間を律速する。
- **③ このリポジトリでの実例**: 実験05で5本の調査を2分割して並列実行 → 遅い方(3本担当)が律速。

## 3. コストと分割判断（分割は常に得ではない）

- **① 一言定義**: sub-agent は毎回**コールドスタート**（指示と文脈を取り直す）＝固定オーバーヘッド。
- **② なぜ必要か・事故例**: 小さいタスクや不均衡な分割では、オーバーヘッド ＞ 恩恵となり、
  トークンが増えるのに時間は縮まない。さらに親が結果を**統合**する手間も乗る。
- **③ このリポジトリでの実例**: 実験05では分割が **約1.45倍のトークンで時間短縮ゼロ**。
  分割が得なのは「各サブタスクが**大きく・独立・均衡・統合が軽い**」とき。

---

## まとめ

sub-agent は強力だが、**分割は既定で得ではなく設計判断**。
「並列だから速い・安い」は幻想で、コスト・均衡・調整を見て決める。
第3部（拡張）はここで一区切り。第4部は既存コードベースへの**運用**（追加開発の自律ループ）へ。
