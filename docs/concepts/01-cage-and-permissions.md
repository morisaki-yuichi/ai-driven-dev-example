# 概念解説 01 — 檻と権限（AI に手足を与える前に）

> 第1部・檻と土台の本編（権限まわり）。執筆日 2026-07-05 / モデル claude-opus-4-8。
> 3点形式：**①一言定義 ②なぜ必要か・事故例 ③このリポジトリでの実例**。

第0部で見たとおり、AI の出力はそのままでは信頼できない。
その AI に「コードを書く」だけでなく「コマンドを実行する」「ファイルを消す」権限まで
与えるなら、**間違いや幻覚が実害に変わる経路**をあらかじめ塞いでおく必要がある。
検証網（前節）が「出力の正しさ」を守る装置なら、檻と権限は「操作の被害範囲」を守る装置。

## 1. ブラスト半径（blast radius）

- **① 一言定義**: ある操作が失敗・暴走したときに**被害が及ぶ最大範囲**。
- **② なぜ必要か・事故例**: AI に広い権限を渡すほど、1回の幻覚・誤操作で壊せる範囲が広がる
  （例: `rm -rf` の対象を取り違える、本番 DB に接続したまま破壊的スクリプトを流す）。
  「起きない」を祈るのではなく、**「起きても被害がここで止まる」を設計**する。
- **③ このリポジトリでの実例**: 開発を **DevContainer（作り直せる容器）** の中で行い、
  ホスト環境やホストの秘密情報に手が届かないようにする。
  → [`.devcontainer/devcontainer.json`](../../.devcontainer/devcontainer.json)。

## 2. 最小権限（least privilege）

- **① 一言定義**: 作業に**必要な最小限の権限だけ**を与え、それ以外は既定で確認/拒否とする原則。
- **② なぜ必要か・事故例**: 「とりあえず全部許可」は楽だが、幻覚コマンドや外部から混入した
  指示（第3部の tool poisoning）がそのまま実行される。危険・外向きの操作ほど、
  自動実行ではなく**人間の確認**を挟むべき。
- **③ このリポジトリでの実例**: Claude Code の権限設定で、**安全と分かっている検証コマンドだけを
  自動許可（allow）**し、破壊的・外向きの操作（`rm`・`git push` 等）は既定のまま確認を要求する。
  → [`.claude/settings.json`](../../.claude/settings.json)。

  ```jsonc
  // .claude/settings.json（このリポジトリの実物・抜粋）
  {
    "permissions": {
      "allow": ["Bash(ruff:*)", "Bash(mypy:*)", "Bash(pytest:*)", "Bash(git add:*)", "…"]
      // rm や git push はここに「無い」＝毎回確認される。
      // 壊す/外に出す操作を自動化しない、が最小権限の肝。
    }
  }
  ```

  常に禁止する `deny`、常に確認する `ask` も同じルール構文で書ける。
  **正確な構文はバージョンで変わりうるため、必ず公式ドキュメントで確認する**
  （知識の鮮度の原則）→ https://docs.claude.com/en/docs/claude-code

## 3. sandbox / DevContainer（檻そのもの）

- **① 一言定義**: 開発環境を**隔離された容器**に閉じ込め、外（ホスト・ネットワーク・秘密情報）への
  アクセスを制限する仕組み。sandbox はコマンド単位、DevContainer は環境単位の隔離。
- **② なぜ必要か・事故例**: 隔離が無いと、AI の誤操作がそのまま**開発マシン全体**へ波及する。
  容器の中なら、最悪でも「容器を捨てて作り直す」で復旧できる。
- **③ このリポジトリでの実例**: `.devcontainer/` に Python 3.12 + Node の隔離環境を定義し、
  検証網ツール（ruff/mypy/pytest）を `postCreate` で導入する。写経者はこの容器の中で
  AI に作業させ、ブラスト半径を容器内に限定する。

## まとめ — 檻と検証網はワンセット

- **検証網**（前節）= 出力の**正しさ**を守る（テスト/型/lint/CI）。
- **檻と権限**（本節）= 操作の**被害範囲**を守る（最小権限・DevContainer）。

片方では足りない。正しさだけ守っても広い権限があれば誤操作で壊れ、
被害範囲だけ絞っても検証が無ければ静かに間違ったものが積み上がる。
第1部はこの2つを土台として敷いた。第2部からは、この土台の上で
「AI に正しい**文脈と仕様**をどう渡すか」に進む。
