# 概念解説 03 — コンテキストの寿命管理（記憶をどこに置くか）

> 第2部の本編。3点形式。執筆日 2026-07-05 / claude-opus-4-8。
> 前提: 概念00「コンテキスト窓」（窓には上限がある）。

窓に上限がある以上、長い作業では必ず「古い文脈が落ちる」瞬間が来る（圧縮／要約）。
だから問うべきは「どれだけ覚えるか」ではなく **「何をどこに置くか」**。

## 1. 4層の記憶モデル

- **① 一言定義**: AI が使える記憶を、寿命と読込タイミングで4層に分ける。
  - **CLAUDE.md（常時文脈）**: 毎回読む。プロジェクトの流儀。
  - **skills（必要時読込）**: 関連する時だけ展開。第3部で扱う。
  - **会話（揮発）**: 今の窓にある間だけ。圧縮で失われる。
  - **リポジトリのファイル（永続）**: 明示的に読めばいつでも戻る。
- **② なぜ必要か・事故例**: どの層に何を置くか決めないと、重要な決定を「会話」に置いてしまい、
  圧縮で消え、後半で方針が食い違う。
- **③ このリポジトリでの実例**: 決定は [`docs/session-logs/`](../session-logs/)（永続）、
  流儀は [`CLAUDE.md`](../../CLAUDE.md)（常時）に置く。

## 2. 決定はファイルへ（会話を信頼の源泉にしない）

- **① 一言定義**: 覚えておくべき決定・制約は、会話に残さずファイルに書く原則。
- **② なぜ必要か・事故例**: 「さっき合意したはず」は圧縮で蒸発する。人間チームが議事録を残すのと同型。
- **③ このリポジトリでの実例**: セットアップ合意（session 000）を会話でなくファイルへ外部化し、
  [実験03](../../experiments/03-context-compaction/) で「今も取り出せる」ことを実行で確認。

## 3. 圧縮（compaction）の扱い

- **① 一言定義**: 窓が一杯になると、古い会話が要約に畳まれる仕組み。細部は落ちる。
- **② なぜ必要か・事故例**: 圧縮の存在を知らないと「なぜ急に前提を忘れた？」に戸惑う。
- **③ このリポジトリでの実例**: 圧縮に備え、①決定をファイル化 ②圧縮時に残す情報を指示、の2手で守る。

## まとめ

- コンテキスト窓（概念00）＝制約。寿命管理（本節）＝その制約下での設計。
- **会話は揮発、ファイルは永続。だから決定はファイルへ。**
