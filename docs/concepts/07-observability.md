# 概念解説 07 — 可観測性（後から再構成できる）

> 第6部。3点形式。執筆日 2026-07-05 / claude-opus-4-8。前提: 概念06「hooks」。

hook（第5部）が**決定的な足場**なら、可観測性はその足場が吐く記録を**読んで再構成する**層。
AI の作業は非決定でその場限り。何を・何回・どの順で・どこに時間がかかったかを計上できなければ、
「なぜ遅い/失敗したか」を語れず、最適化もデバッグもできない。

## 1. トレース（行動の記録）

- **① 一言定義**: ツール呼び出しを1行1レコードで残し（JSONL）、後から時系列に再構成できる形。
- **② なぜ必要か・事故例**: 記録が無いと、遅さ・失敗は「体感」でしか語れない。どのステップが原因かを
  当てずっぽうで探すことになる。**観測できないものは最適化できない**。
- **③ このリポジトリでの実例**: PostToolUse hook [`trace.py`](../../.claude/hooks/trace.py) が
  `docs/traces/tool-calls.jsonl` に追記し、[`tools/trace_summary.py`](../../tools/trace_summary.py) が読む。
  hook → JSONL → 集計の**一本道**（[実験07](../../experiments/07-observability/)）。

## 2. 何を計上するか（行動・時間・コスト）

- **① 一言定義**: 少なくとも**ツール別の回数**・**タイムライン（期間と遅いステップ）**。可能なら**コスト**。
- **② なぜ必要か・事故例**: 内訳が見えると「Bash が54%」「`#3 pytest` が突出して遅い」と標的が定まる。
  逆にコスト（トークン/価格）は**hook 入力に無い**——ここを推測で埋めると事故る（知識の鮮度）。
- **③ このリポジトリでの実例**: 集計は counts / span / 「遅かったステップ」を出す。**トークン/コストは別系統**
  （OpenTelemetry の usage メトリクス等）とし、第7部 eval のモデル階級比較で回収する Try に置いた。

## 3. 観測の正直さ（測れているものを正しく名指す）

- **① 一言定義**: 出した数字が**実際に何を測っているか**を偽らないこと。近似は近似と明記する。
- **② なぜ必要か・事故例**: 「ステップ経過」を「ツール単体レイテンシ」と偽ると、間違った所を最適化する。
  隣接 `ts` の差分はモデル思考や**人間の待ち**（`AskUserQuestion`）も含む。実験07では
  人間が出荷可否を決めた 127s がそのまま1ステップとして現れた。
- **③ このリポジトリでの実例**: 集計器はラベルを「前ステップからの経過＝ツール＋モデル＋待ち込み」とし、
  async 追記で混じる**壊れた行**は行単位で飛ばして数を報告。純粋な単体時間は Pre/Post ペア計測（Try）と明記。

---

## まとめ

可観測性は「行動・時間を確実に計上し、正直に名指す」層。hook（決定的な足場）が記録を生み、集計が
それを**後から再構成できる**知識に変える。守る集計器自身も検証網に載せる
（[`test_trace_summary.py`](../../tools/test_trace_summary.py)・gate に `tools` を追加）。
次は第7部——この土台（行動・時間・コスト）の上で、**変更が本当に良くなったかを測る**eval へ。
