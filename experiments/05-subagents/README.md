# 実験05 — sub-agent 分割は常に得ではない（委譲・並列・コスト）

> 第3部の実験。形式は 予想 → 観察（実データ）→ 結論（設計指針）。
> 記録日 2026-07-05 / claude-opus-4-8 / sub-agent は Explore。
> 関連: [`concepts/05`](../../docs/concepts/05-subagents.md)。

## ねらい

「大きな調査を1本 vs sub-agent 分割」を**実起動**して、時間・コスト・質を比較する。
分割（並列）は直感的に速そうだが、常に得とは限らないことを実データで確かめる。

## タスク（両者同一）

実験00〜04（5本）の README を読み、各実験の予想/観察/検出網を1〜2行に要約するカタログ作り。

## 予想

分割すれば並列で速くなる。ただし sub-agent はコールドスタート（毎回、指示と文脈を取り直す）
＝固定オーバーヘッドがあるので、小さいタスクや不均衡な分割ではトークンが増え、時間短縮も出にくい。

## 観察（実データ・各1回・Explore）

| 方式 | 構成 | 所要時間 | トークン | 備考 |
|------|------|---------:|---------:|------|
| 単一 | 1エージェントで5本 | 25.9s | 16,446 | 統合不要 |
| 分割A | 実験00-02 | 27.0s | 13,459 | |
| 分割B | 実験03-04 | 18.2s | 10,466 | |
| **分割 計** | 2エージェント並列 | **≈27.0s**（遅い方Aで律速） | **23,925** | ＋親側の統合コスト |

- **時間**: 分割 ≈27.0s ≧ 単一 25.9s。**速くならなかった**。遅い shard（A＝3本）が全体を律速し、
  per-agent の固定コストが並列の恩恵を食った。
- **コスト**: 分割 23,925 tok は単一 16,446 tok の **約1.45倍**。さらに親が2出力を統合する分だけ増える。
- **質**: どちらも正確。ただし分割は shard 間で**体裁がずれた**（A は `**見出し**`、B は `## 見出し`）ため、
  親が整える手間（＝調整コスト）が要る。

## 結論（いつ分割が得か）

分割・並列が得なのは **①各サブタスクが大きい ②互いに独立 ③均衡している ④統合が軽い** とき。
今回は小さく・不均衡・統合ありだったので、**分割は損**（約1.45倍のコストで時短ゼロ）。
**「並列だから速い・安い」は幻想**。分割は設計判断であり、既定で得ではない。

## 正直さについて

各方式1回の測定（非決定性で数値は揺れる）。固定できるのは数値の絶対値ではなく、
**「sub-agent には固定オーバーヘッドがあり、小さい/不均衡なタスクでは分割が損になりうる」**という構造。
時間はコールドスタート込みの実測、トークンは usage メタデータの実数。
なお最初の単一は同期実行で usage が取れなかったため、公平を期して同条件（背景実行）で1回測り直した。
