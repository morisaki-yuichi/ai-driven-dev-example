# 実験01 — ブラスト半径の封じ込め（檻の意味）

> 第1部（檻と土台）の実験。形式は **予想 → 観察 → 封じ込めの確認**。
> ※ 第0部の「予想→観察→検出網」の3拍子を、檻系の実験では
> 「観察→**被害の限定（＝檻の壁）**」に読み替える（検出ではなく封じ込めが主題のため）。
> 実行環境: bash / 記録日 2026-07-05 / 著者モデル claude-opus-4-8。
> 関連: [`docs/concepts/01-cage-and-permissions.md`](../../docs/concepts/01-cage-and-permissions.md)。

## ねらい

AI に破壊的な操作の権限を渡すと、幻覚や誤解が**実害**に変わりうる（概念01「ブラスト半径」）。
その被害を **容器（DevContainer/sandbox）の中に閉じ込める**——「起きても、ここで止まる」を
体験する。**本物のファイルは一切壊さない**：使い捨ての `mktemp` 領域だけを対象にする。

## 予想

AI が「後片付けしておいて」を破壊的に解釈し、作業ディレクトリで `rm -rf ./*` を暴走させる。
だが作業が**容器の中**で行われていれば、消えるのは容器の中身だけで、
容器の外（ホスト＝本物のプロジェクト）は無傷のはず。

## 手順（自分でやってみる）

```console
bash experiments/01-blast-radius/demo.sh
```

スクリプトは容器役とホスト役を `mktemp -d` で2つ作り、容器の中でだけ破壊を起こし、
最後に容器ごと破棄する（`trap ... EXIT`）。

## 観察（実際の出力）

```console
$ bash experiments/01-blast-radius/demo.sh
=== 破壊前 ===
[容器の中]      : app.py important_data.txt
[容器の外/ホスト]: host-secret.txt

=== AI が『後片付けして』を破壊的に解釈して暴走 → 容器の中で rm -rf ./* ===

=== 破壊後 ===
[容器の中]      : ''  ← 空（全滅）
[容器の外/ホスト]: host-secret.txt   ← 無傷
```

さらに、実験の前後で**本物のリポジトリ**を確認したが変化なし（`README.md` 健在、
`git status` は実験前と同一）。破壊は容器の中で止まった。

## 封じ込めの確認（＝檻の壁）

| 対象 | 破壊コマンドの影響 |
|------|--------------------|
| 容器の中（AI の作業場） | 全滅（想定どおり） |
| 容器の外（ホスト見立て） | 無傷 |
| 本物のリポジトリ | 無傷（README・git 状態が実験前と同一） |

被害範囲＝ブラスト半径が「容器の中」に限定された。これが檻の効果。
最悪でも「容器を捨てて作り直す」で復旧できる。

## 教訓

1. **検証網は「正しさ」、檻は「被害範囲」を守る**——役割が違い、両方要る（概念01）。
2. **破壊的・外向きの操作は自動化しない**——最小権限（[`.claude/settings.json`](../../.claude/settings.json)）で
   `rm` や `git push` を allow に入れず、毎回確認させる。今回は「起きた場合」に容器が受け止めた。
3. 檻があるからこそ、AI に手足（コマンド実行）を安心して渡せる。

## 正直さについて（この実験の作り方）

ここでの「容器」は `mktemp` の使い捨てディレクトリで、**真の隔離ではなく見立て**である。
実際の DevContainer/sandbox は OS/コンテナ境界による、より強い壁を提供する
（ホストのファイルシステムやネットワーク自体に手が届かない）。本実験は
「被害を作業場の外に出さない」という**発想**を、依存なしで安全に体験するためのもの。
真の隔離の検証は `.devcontainer/` を実ビルドした環境で行う。
